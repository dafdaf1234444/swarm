# L-1207: Pre-commit hook duration creates HEAD race window — third concurrent git failure mode
Session: S476 | Domain: meta, catastrophic-risks | Sharpe: 8 | Level: L3
Cites: L-969, L-970, L-971, L-526, L-601, L-688
Confidence: Observed (n=4 failures in single session, 3 distinct mechanisms)

## Finding
At N>=5 concurrent sessions, three distinct git failure modes interact: (1) index corruption — concurrent writes produce "index file smaller than expected" (L-969); (2) mass-staging cascade — corrupted index shows all files as deleted (L-971); (3) HEAD race — pre-commit hook takes ~20s (maintenance.py runs 16 periodics), during which concurrent sessions advance HEAD, causing "cannot lock ref HEAD" on commits that passed all checks. GIT_INDEX_FILE workaround (L-970) fixes (1)+(2) but not (3). Only fix for (3): reduce hook duration. check.sh --quick still runs full maintenance.py. A --minimal mode (guards only, no maintenance) would cut hook time from ~20s to ~3s.

## Rule
At high concurrency, use GIT_INDEX_FILE=/tmp/temp-index for index isolation. For HEAD races: minimize pre-commit hook duration. Structural fix: add check.sh --minimal (guards only, skip maintenance.py).

Falsified if: check.sh --minimal doesn't reduce HEAD race failure rate at N>=5 concurrency.
