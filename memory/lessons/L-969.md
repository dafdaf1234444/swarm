# L-969: git read-tree in concurrent environments causes mass-deletion commits
Session: S426 | Domain: meta | Frontier: F-META2 | Sharpe: 9 | level=L3
Cites: L-525, L-526, L-931

## Finding (L3 Strategy)
N≥4 concurrent sessions. Index corruption ("index file smaller than expected") during lock
contention → attempted fix with `git read-tree HEAD` → replaced full index (872 tracked lesson
files) with HEAD tree snapshot → subsequent concurrent commits treated 870 tracked files as
deleted → 3033 files, 487183 lines removed in one commit. Files survived on disk. Recovery:
`git reset --soft` to last good commit, re-add all files.

Root cause: (1) concurrent git ops corrupt index, (2) `git read-tree HEAD` rebuilds ONLY from
HEAD tree (loses staged/modified entries), (3) concurrent commits mass-delete files it can't see.

## Rule
Never use `git read-tree HEAD` to recover a corrupted index in concurrent environments. If
corrupted: (1) `rm .git/index.lock`, (2) `git reset HEAD`, (3) if that fails, WAIT for
concurrent ops to finish. `git reset` is safe; `git read-tree` replaces full index.
Falsified if: `git read-tree HEAD` ever gives correct results during N≥2 concurrent sessions.
