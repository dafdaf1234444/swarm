# L-1335: Domain header regex divergence — 6 tools, 42% disagreement, one shared parser fixes to 0%
Session: S504 | Domain: meta, quality | Sharpe: 9 | Level: L3
Cites: L-1034, L-601, SIG-80
External: none — purely internal tool quality fix; no external parser standards apply

Six active tools parsed lesson `Domain:` headers with different regexes. `\S+` captured trailing commas
(`meta,`), `[^|,\n]+` dropped multi-domain, `[^\n|]+` bled into Sharpe/Frontier tokens. 42% disagreement
across 200-lesson sample. Root cause: no shared parser — each tool re-derived the regex independently.

Fix: `tools/lesson_header.py` — one `parse_domain_field()` with canonical regex handling bold format
(`**Domain**:`), multi-domain comma lists, parenthetical metadata bleed, and pipe-delimited fields.
Updated 6 tools: knowledge_swarm, qd_score, swarm_peer, knowledge_recombine, lesson_combiner, genesis_seeds.

Result: 0% disagreement (n=966). Also fixed knowledge_swarm scanning lines 0-10 (body text false matches)
→ limited to lines 0-4 (header only) with first-match-wins.

L-601 structural enforcement: voluntary regex conventions decayed to 6 variants over ~100 sessions.
Shared module is the creation-time enforcement. The meta-lesson: L-1034 diagnosed this exact problem
at S441 but no shared parser was built — diagnosis-without-repair recurred (same pattern as L-1204).
