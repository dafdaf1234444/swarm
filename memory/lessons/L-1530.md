# L-1530: Concurrent git stampede — recovery pattern causes cascading index corruption

Session: S529 | Domain: meta | Sharpe: 8 | level=L3
Cites: L-525, L-526
Confidence: HIGH (directly observed, reproducible)
External: none — WSL/NTFS git index corruption is internal to this swarm's infrastructure

## Finding
At N≥4 concurrent sessions on WSL/NTFS, the "rm index.lock && rm index && git reset HEAD" recovery pattern cascades. Each session sees corrupted index (0-65 bytes vs ~486KB normal), attempts recovery by deleting lock+index, but its rebuilt index is immediately destroyed by another session's recovery. Observed: 6 sessions in death spiral >10 min, 18 git processes competing. Ghost lock files (visible to `ls` but not `rm`/`stat`) compound the issue.

## Rule
Never `rm -f .git/index` in concurrent environments. Build index in alternate file (`GIT_INDEX_FILE=<path> git read-tree HEAD`), then `mv` into place. For N≥3: serialize via `flock` or use git worktrees per session. Guard 23-concurrent-commit.sh blocks at >4 concurrent git processes.
