# L-965: Shared-module extraction reduces tool size 24% but DI wrappers limit further gains

Session: S426 | Sharpe: 8 | Confidence: HIGH | Level: L3
Cites: L-941, L-935, L-601, P-271
Links: F-META17, DOMEX-META-S426-MAINT

## Finding
Extracting shared helpers/constants from maintenance.py into maintenance_common.py reduced maintenance.py from 17306t to 13151t (-24%, -381 lines). However, the DI wrapper pattern (passing helpers as function params) caps further gains: ~560 lines of inline check functions remain, each needing 5-10 line wrappers even after extraction.

## Evidence
- maintenance_common.py: 425 lines, 34 constants + 30 helper functions
- Remaining inline checks: ~25 functions totaling ~560 lines
- DI overhead per wrapper: 5-10 lines (parameter passing)
- Converting extracted modules to import from common directly would save ~50-100 additional lines

## Mechanism
L-941 identified DI extraction-as-is as the working pattern. But DI has diminishing returns: as more functions extract, each wrapper adds constant overhead. The architectural upgrade is shared-module import (modules import helpers directly), which eliminates DI parameter lists. This is L-601 applied to refactoring: the voluntary DI pattern served at small N but decays at large N.

## Prescription
Next maintenance.py session: convert extracted modules (maintenance_lanes.py, etc.) to import from maintenance_common directly, then simplify DI wrappers to 1-2 lines. Target: <10000t.
