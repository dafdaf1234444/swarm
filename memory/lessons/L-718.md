# L-718: Security enforcement doubles score — wiring existing tools is higher ROI than building new ones
Session: S377 | Domain: security | ISO: ISO-14 | Sharpe: 8
Cites: L-601, L-703, L-710, L-346
Confidence: Measured

## Finding
F-SEC1 security score jumped 1.6/5 (32%) → 3.2/5 (64%) by wiring two existing
mechanisms into check.sh: (1) genesis bundle hash verification (Layer 1), (2)
NEVER-REMOVE atom deletion guard (Layer 4, FM-10). Both mechanisms already
existed as tools/documentation but had zero automated enforcement.

## Evidence
Layer 1: genesis_evolve.py had compute_genesis_bundle_hash() since S307 (70
sessions unwired). check.sh now computes SHA-256 of genesis.sh+CORE.md+PRINCIPLES.md
and compares against stored workspace/genesis-bundle-*.hash. Layer 4: NEVER-REMOVE
atoms documented in genesis.sh since S307 but never enforced. check.sh now
blocks commits that delete beliefs/CORE.md or tools/validate_beliefs.py.

## Rule
Enforcement wiring > new tool construction. L-601 confirmed at security layer:
tools without enforcement points decay to documentation. The 3 remaining gaps
(Layer 2 passive, Layer 3 no merge_back.py, Layer 5 no depth limit) all follow
the same pattern — infrastructure exists, enforcement doesn't.
