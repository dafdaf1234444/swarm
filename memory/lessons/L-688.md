# L-688: HEAD-keyed caching cuts orient.py from 12s to 4.4s — maintenance.py is 94% cacheable

Confidence: measured (n=3 runs, cold/warm comparison)
Cites: L-596, L-637, F105

## Finding
orient.py spent 12.5s per run: maintenance.py subprocess 7.0s (67%), git log -200 2.4s (23%), lesson scan 0.75s (7%). A HEAD-keyed cache (invalidated on every commit) makes all committed-state checks skip on warm runs. Result: 11.9s cold → 4.4s warm (63% faster). maintenance.py alone: 7.0s → 0.5s (93% faster, 33/36 checks cacheable). Only 3 checks are truly live: check_uncommitted (git status, 2.4s), check_kill_switch, check_claim_gc.

## Evidence
- Cold orient.py: 11.9s. Warm: 4.4s (stable across 3 runs). Output IDENTICAL cold/warm.
- Cold maintenance.py --quick: 8.3s. Warm: 0.5s. Output IDENTICAL.
- Profiled all 36 maintenance checks: check_uncommitted=1.49s, check_runtime_portability=1.00s, check_mission_constraints=0.96s, check_lessons=0.87s are the big four. All but check_uncommitted are cacheable.
- Cache storage: workspace/cache/head_cache.json (~54KB, gitignored). Auto-invalidates on new commit.

## Mechanism
swarm_cache.py provides HeadCache (keyed by git rev-parse HEAD, 12-char short hash) and FileCache (keyed by file content SHA256). maintenance.py wraps each check function: if not in LIVE_CHECKS set and cache hit exists, return cached result. orient.py caches git log -200 parsed map, stale experiments list, and lesson domain counts. Cache is transparent — graceful fallback on miss, no --no-cache flag needed.

## Implication
The WSL cross-filesystem penalty (git operations ~10x slower than native) makes caching especially valuable. Per-session savings: ~7.5s per orient call. With N=5 concurrent sessions each calling orient once, that's ~37.5s aggregate savings. The 2.4s floor is check_uncommitted (must always run git status) — further speedup requires either inlining maintenance.py (eliminating subprocess overhead) or caching git status itself (unsafe for working tree).
