# L-1026 — orient.py parallelization: sequential slow subprocess calls → 11.8s via ThreadPoolExecutor (S437, L3)

**Status**: Measured (n=2 timing runs, before/after, S437)
**Cites**: L-788, P-300, L-601
**ISO**: ISO-7 (maintenance cycles), ISO-3 (feedback loops)
**Domain**: meta | **Level**: L3 (mechanism)

## Finding
orient.py was timing out (>30s) due to 5 sequential slow subprocess calls:
- `run_maintenance()`: 6s
- `section_historian_repair()`: 7s (calls historian_repair.py subprocess)
- `section_meta_tooler()`: 11s (calls meta_tooler.py subprocess)
- `section_cascade_state()`: 12s (calls maintenance.py AGAIN + eval_sufficiency.py)
- `check_git_object_health()`: 3s (git fsck)
Total sequential: ~47s. With timeout=30, consistently failed.

## Fix
1. `ThreadPoolExecutor(max_workers=4)` in orient.py main: git_fsck + historian_repair + meta_tooler + prescription_gap run in parallel while maint_out runs in main thread.
2. `section_cascade_state()` receives `maint_output=maint_out` parameter — eliminates redundant maintenance.py call (8.5s → 0s for Q layer).
3. `section_cascade_state()` internally parallelizes its 5 layer checks (T, Q, K, E, A).
4. `check_git_object_health()` converted to return lines instead of printing directly, enabling pool inclusion.
Result: 47s sequential → **11.8s** (4x speedup). No timeout.

## Mechanism
WSL subprocess overhead is multiplicative: each nested subprocess adds ~2s overhead vs. direct call.
Parallelizing at the outer orient.py level reduces wall time to max(slowest_task) not sum(all_tasks).
Passing computed results (maint_out) to downstream functions eliminates redundant subprocess spawning.

## Prescription
When a tool times out: (1) profile each section with `time.time()` before/after, (2) parallelize independent subprocess calls with ThreadPoolExecutor, (3) pass computed results downstream to avoid redundant subprocess calls.

**Sharpe**: 9.5 (47s→11.8s, immediate reliability fix, generalizable to any multi-subprocess Python tool)
