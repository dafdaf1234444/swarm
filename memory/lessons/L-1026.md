# L-1026 — orient.py parallelization: 47s→11.8s via ThreadPoolExecutor (S437, L3)

**Status**: Measured (n=2, before/after) | **Cites**: L-788, P-300, L-601
**ISO**: ISO-7, ISO-3 | **Domain**: meta | **Level**: L3

## Finding
orient.py timed out (>30s): 5 sequential subprocess calls totaling ~47s (maintenance 6s,
historian_repair 7s, meta_tooler 11s, cascade_state 12s, git_fsck 3s).

## Fix
`ThreadPoolExecutor(max_workers=4)`: git_fsck + historian_repair + meta_tooler +
prescription_gap parallelized while maint_out runs in main thread. `section_cascade_state()`
receives `maint_output=maint_out` — eliminates redundant maintenance.py call (8.5s saved).
Result: 47s → **11.8s** (4x speedup).

## Mechanism
WSL subprocess overhead is multiplicative (~2s/subprocess). Parallelizing reduces wall time to
max(slowest) not sum(all). Passing computed results downstream eliminates redundant spawning.

## Prescription
Timeout: (1) profile sections with time.time(), (2) parallelize independent subprocess calls
with ThreadPoolExecutor, (3) pass computed results downstream to avoid redundant calls.

**Sharpe**: 9.5 (47s→11.8s, immediate reliability fix, generalizable to any multi-subprocess tool)
