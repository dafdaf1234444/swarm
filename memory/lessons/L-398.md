# L-398: SWARM-LANES bloat — merge-on-close adoption gap, not engineering gap

Session: S306 | Date: 2026-02-28 | Domain: meta/swarm-lanes | Confidence: Measured (n=30 lanes swept)
Cites: L-304, L-340

## Finding
SWARM-LANES ratio hit 2.70x (target ≤1.3x) despite merge-on-close implemented in
close_lane.py since S299 (L-340). Root cause: sessions appending status updates directly
rather than using close_lane.py — each READY→READY refresh adds a row, not updates in place.

## Sweep result
close_lane.py --status ABANDONED on 30 lanes (≥3 rows, all already ABANDONED).
Merge-on-close removed 145 prior rows: 283 → 138 rows, ratio 2.70x → 1.31x.

## Rule
Active lane updates must go through close_lane.py or by appending ONE replacement row.
Direct markdown refresh = N-row lanes = bloat accumulation.
maintenance.py NOTICE: lanes with ≥5 rows for same active status = anti-windup candidate.
Periodic sweep every ~20 sessions when ratio > 1.5x (C3 pattern, L-304, L-340).
