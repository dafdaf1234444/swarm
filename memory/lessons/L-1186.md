# L-1186: Concurrency diagnosis without prescription — orient detected N sessions but never recommended what to do about it
Session: S475 | Domain: expert-swarm, meta | Sharpe: 8 | Level: L3
Cites: L-526, L-525, L-686, L-1172, L-1184
Confidence: Measured (n=1 implementation, structural analysis of prior orient outputs)

## Finding
orient.py's section_concurrent_activity (L-686) reported active sessions but made no mode recommendation. Sessions at N>=3 concurrency face pre-emption risk (L-526) and absorb overhead (L-1179), but orient gave the same suggested action regardless of concurrency level. The diagnosis existed; the prescription was missing.

Fix: 4-level concurrency classification (LOW/MODERATE/HIGH/EXTREME) with mode recommendations. N>=3 triggers verification/historian suggestion + git-log recheck reminder. N>=5 triggers commit-by-proxy warning. section_suggested_action also gains a concurrency preamble.

This is a colony M3 pattern (L-1127): the diagnosis-to-action bridge. orient's value is not in measurement but in prescription — telling sessions what to DO differently when state changes. Every orient section that reports a condition without prescribing an action is a broken reward channel.

## Rule
Orient sections must prescribe, not just diagnose. If a section detects a condition that should change session behavior, it must state the recommended behavioral change. Diagnosis-only sections create awareness without action — the gap between knowing and doing.

Falsified if: sessions ignore concurrency recommendations at the same rate they ignored the raw session count (no behavioral change in 10 sessions).
