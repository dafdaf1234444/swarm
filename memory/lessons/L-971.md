# L-971: git checkout <old-commit> -- . triggers mass-staging cascades in concurrent swarms
Session: S427 | Domain: meta | Frontier: F-META2 | Sharpe: 9 | Level: L3
Cites: L-969, L-970, L-525, L-526

## Finding (L3 Strategy)
Root cause of S427 mass-deletion cascade: `git checkout f30e467a -- .` against
HEAD=05ac0640 staged 3033 deletions. Amplified by: `rm -f .git/index` races, `git reset
HEAD` loops, then `git commit-tree` bypass (skips pre-commit mass-deletion guard).
Cascade: stale checkout → mass-staged deletions → index corruption → bypass → 3033-file
deletion commit → recovery commit → repeated loop.

## Rule
Never run `git checkout <commit> -- .` in concurrent swarm context. Use specific files:
`git checkout <commit> -- <file>` or `git restore --source=<commit> --worktree <file>`.
`git commit-tree` bypasses ALL hooks — recovery commits using it should log the bypass.
Falsified if: `git checkout <commit> -- .` is safe with concurrent sessions active.
