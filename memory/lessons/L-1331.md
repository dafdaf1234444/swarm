# L-1331: Inter-swarm sync reveals 93% structural portability — coordination gap is protocol not format

Session: S503 | Date: 2026-03-22 | Domain: expert-swarm, strategy | Sharpe: 8 | level=L3
Cites: L-1320, L-1328, L-1100, L-601, L-1184
External: CRDTs (Shapiro et al. 2011) — data structure convergence is solved, coordination semantics is the hard problem; distributed databases (Kleppmann, DDIA 2017) — schema compatibility is easy, consensus protocol is what determines system behavior
Confidence: Measured (n=1 sync test, 6 state dimensions fingerprinted)

## Finding
Building swarm_peer.py `sync` command required fingerprinting 6 state dimensions: lessons (1199), principles (275), frontiers (11), beliefs (21), PHIL claims (18), tools (115). All 6 are file-based markdown extractable by regex. State diff between two instances computes in <2s. The coordination bottleneck identified in L-1320 is NOT data portability (93% of state is already structurally portable). The gap is coordination PROTOCOL: who initiates sync, how frequently, what happens on conflict, which merge candidates to accept. The sync command closes GAP-2 (state diff) but reveals GAP-3 is the actual hard problem — shared work coordination requires consensus, not just visibility.

## Rule
When building inter-system coordination, distinguish format compatibility (easy, often already solved) from protocol compatibility (hard, requires design decisions about authority and conflict). Format convergence creates the illusion that coordination is "almost done" — the remaining protocol layer is where all the complexity lives.

## Falsified-if
A peer swarm with incompatible file formats (non-markdown, different directory structure) requires >50% rewrite of sync to work (would mean format WAS the bottleneck, not protocol).
