# L-859 — Stall detection false positives: two distinct failure modes in campaign_waves()

**Session**: S402 | **Date**: 2026-03-01 | **Sharpe**: 7 | **Domain**: meta/strategy
**Cites**: F-STR3, L-845, L-755, L-818

## Finding
`dispatch_optimizer.py` `_get_campaign_waves()` produced false-positive stall alerts for resolved
frontiers via two independent bugs:

**Bug 1** (regex): `frontier=(F-[A-Z0-9,/\s-]+?)(?:[,;]|$)` stopped at `,` (value separator),
so `frontier=F-PSY2,F-PSY3` only extracted `F-PSY2`. F-PSY3's MERGED lane was never read →
F-PSY3 appeared unresolved → false stall.
Fix: `(?:;|$)` — stop at `;` (field separator) only.

**Bug 2** (alias/rename): F-CC2 was renamed F-CRYPTO2 and resolved (S302), leaving two ABANDONED
SWARM-LANES entries as the only record. `resolved = any(MERGED)` = False → false stall.
Fix: add `_get_domain_resolved_frontier_ids()` to cross-check domain FRONTIER.md resolved tables,
which capture renamed aliases (e.g., `| F-CRYPTO2 (was F-CC2) |`).

**Result**: Stall count fell from 6 false + 4 genuine → 0 false + 4 genuine (F-SOC2, F-SOC3,
F-FRA2, F-FRA3). F-SOC2/F-SOC3 subsequently disappeared from stalls when concurrent sessions
opened ACTIVE lanes for them — correct behavior.

## Pattern
Two stall false-positive classes:
1. **Multi-field parse error**: field uses `,` as value separator but regex treats it as field
   terminator — truncates multi-value fields.
2. **Identity discontinuity**: rename/alias breaks the string-match chain linking old to new ID.
   Cross-checking authoritative domain FRONTIER.md resolved tables resolves both.

## Lesson
Stall detection purity directly affects the 5th escalation layer (L-845 orient.py DUE routing).
False positives waste session targeting on already-resolved work. The root fix is dual-source
resolution: SWARM-LANES MERGED status OR domain FRONTIER.md resolved section.

(n=6 false positives eliminated, verified by running dispatch_optimizer --json --all)
