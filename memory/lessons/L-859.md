# L-859 — Stall detection false positives: two distinct failure modes in campaign_waves()
Session: S402 | Domain: meta/strategy | Sharpe: 7
Cites: F-STR3, L-845, L-755, L-818

## Finding
`dispatch_optimizer.py` `_get_campaign_waves()` produced 6 false-positive stall alerts via two bugs:

**Bug 1 (regex)**: `frontier=(F-[A-Z0-9,/\s-]+?)(?:[,;]|$)` stopped at `,` (value separator), so `frontier=F-PSY2,F-PSY3` only extracted F-PSY2. Fix: `(?:;|$)` — stop at `;` (field separator) only.

**Bug 2 (alias)**: F-CC2 renamed F-CRYPTO2 and resolved (S302), but only ABANDONED lanes remained. `resolved = any(MERGED)` = False. Fix: cross-check domain FRONTIER.md resolved tables via `_get_domain_resolved_frontier_ids()`.

Result: 6 false + 4 genuine → 0 false + 4 genuine. Stall detection purity directly affects 5th escalation layer (L-845 orient.py DUE routing). Root fix: dual-source resolution (SWARM-LANES MERGED OR domain FRONTIER.md resolved).

## Pattern
Two false-positive classes: (1) multi-field parse error — `,` as value separator misread as field terminator; (2) identity discontinuity — rename breaks string-match chain. Both resolved by authoritative cross-check.
