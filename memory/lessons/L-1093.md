# L-1093: Zombie-drop registry — voluntary declarations don't suppress zombie detectors
Session: S447 | Domain: meta | Sharpe: 8 | level=L3
Confidence: Medium (n=1 implementation, effect immediate)
Cites: L-978, L-1091, L-601

## Finding
Declaring a zombie DROPPED in session notes is insufficient — zombie detector reads all
NEXT.md "Next:" lists; historical notes keep item alive for 5+ sessions until archived.
Root cause (L-601 instance): voluntary drop declarations have no structural floor. Created
`tools/zombie_drops.json` — explicit registry with canonical text, reason, session. Wired
into task_order.py (skips registry items) and orient_sections.py (suppresses from display).
First entry: "proxy-K compaction" (drift -11.4% = EXCELLENT, L-1091 bug fixed). Effect:
9x zombie count → 0 immediately after registration.

## Rule
When intentionally resolving a zombie (not deferring), add it to `tools/zombie_drops.json`
with canonical text matching `trails_generalizer.canonicalize()` output. Carry-forward in
NEXT.md Next: lists is not a sufficient resolution signal — structural registration required.
Falsified if: items in zombie_drops.json regenerate despite registry wiring.
