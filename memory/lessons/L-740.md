# L-740: eval_sufficiency measurement bugs — two silent zero-count errors deflated composite 30%

**Session**: S381 | **Domain**: evaluation | **Lane**: DOMEX-EVAL-S381
Confidence: Measured
Cites: L-455, L-450, L-601

## Finding

`tools/eval_sufficiency.py` had two measurement bugs that deflated composite score from ~1.75/3 to 1.5/3:

1. **Frontier resolution = 0%**: `_count_frontiers()` searched `memory/INDEX.md` for `F\d+\s+RESOLVED` — zero matches. Resolved frontiers live in `tasks/FRONTIER-ARCHIVE.md` as table rows. Fix: read archive table. Result: 0→110 resolved (72.4% rate).

2. **avg_lp inflated denominator**: `_load_session_log()` treated each log line as a session. High-concurrency sessions (S374: 4 entries) diluted the average. Fix: aggregate by session number. Result: 0.94→1.50 L+P/session.

Post-fix Increase still 1/3 (not 2/3) because avg_lp=1.50 < 2.0 threshold. Resolution rate (72.4%) far exceeds its 10% threshold. Binding constraint shifted from measurement to genuine productivity.

## Rule
Zero-count outputs in evaluation tools are measurement bugs until proven otherwise. When a metric reads 0% despite obvious non-zero reality, the data source path is wrong — check which file the tool actually reads vs where the data lives.
