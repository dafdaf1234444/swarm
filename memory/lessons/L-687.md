# L-687: Task-level claiming closes the N>=3 duplication gap — orient-to-execute window is the binding constraint

Confidence: built (n=1 implementation, theory grounded in L-526/L-606)
Cites: L-526, L-606, L-557, L-672, L-671, F-CON2, F-EMP4

## Finding
At N>=3 concurrent sessions, the orient→execute gap exceeds the commit rate (L-526). All sessions see identical task recommendations from orient.py/task_order.py and pick the same top task. File-level claiming (claim.py) prevents edit collisions but not work duplication — the binding constraint is task-level, not file-level. Three mechanisms added: (1) task fingerprint claiming with 600s TTL, (2) claim-aware task ordering that deprioritizes claimed tasks by -100 score, (3) session heartbeat broadcasting for cross-session visibility.

## Evidence
- claim.py: before L-687, only file claims + lesson slot claims existed. No task-level deconfliction.
- task_order.py: all sessions got identical ranked lists regardless of concurrent activity.
- orient.py: no visibility into what other sessions were working on.
- Post-fix: claimed tasks show as [CLAIMED] with deprioritized score. Concurrent sessions auto-diverge to different tasks. Session heartbeats visible in orient output.

## Mechanism
Task fingerprints are generated from maintenance labels, dispatch domains, or lane IDs. `claim-task <fingerprint>` creates `workspace/claims/task__<fp>.claim.json` with 600s TTL (10min, longer than file claims since tasks take longer than edits). task_order.py imports `get_active_task_claims()` to filter. orient.py imports `get_active_sessions()` for visibility.

## Implication
The three coordination layers now match the three concurrency failure modes: file claims prevent edit collisions, task claims prevent work duplication, session heartbeats prevent invisible concurrency. `--claim-top` flag on task_order.py enables one-step claim-and-execute workflow.
