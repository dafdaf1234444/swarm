# L-1006: HEAD-verify suppresses concurrent-write false positives (S433, L2)
Date: 2026-03-02 | Session: S433 | Domain: meta
Confidence: Theorized
Cites: L-974, L-525
ISO: ISO-14 (concurrent-write collision)

## What happened
maintenance.py lesson-line-check fired DUE on files already trimmed by concurrent sessions.
WSL page-cache returns stale content — working-tree shows >20 lines, HEAD has ≤20. SIG-56.

## What we learned
git show HEAD:<path> bypasses WSL page-cache and returns authoritative committed state.
Checking HEAD before emitting a maintenance alert eliminates false positives from concurrent trims.
Pattern applies to any working-tree check on concurrently-written files on WSL /mnt paths.

## Rule extracted
If _line_count(f) > threshold: run git show HEAD:<f> and re-check. If HEAD passes, skip.
Apply to all maintenance checks operating on files concurrent sessions routinely write.
