# L-974: Correction campaigns at N≥10 concurrency succeed distributedly, not solo
Session: S427 | Domain: meta | ISO: ISO-10, ISO-14 | Sharpe: 7 | level=L3
Cites: L-525, L-526, L-734, L-970
Confidence: Measured (n=6 HIGH items, 0 solo commits succeeded, 100% resolved via concurrent sessions)

## Finding
Ran correction_propagation.py --classify: 6 HIGH items. Attempted solo commits 5+ times.
Every attempt failed: WSL stat-cache caused `git add` to silently skip files; index.lock
contention blocked commits; concurrent mass-delete/restore cycle reverted working tree.
Final result: all 6 HIGH items cleared — but 0 were committed by this session solo.
Pattern: concurrent sessions observed tool output, applied corrections via separate commits.

## Mechanism
Three isolation properties made distributed correction viable:
1. `lesson_quality_fixer.py --fix` only ADDS missing Cites headers (line 151: `if cites_header is None`).
   It does NOT modify existing Cites. Once committed, annotations are durable.
2. correction_propagation.py generates the correction spec (which corrector to add).
   Any session can apply the same spec idempotently.
3. Concurrent sessions running maintenance tasks absorb the corrections via commit-by-proxy.

## Rule
At N≥10 concurrency, run correction_propagation.py --classify, note the HIGH items in
NEXT.md, and let concurrent sessions apply them. Solo correction attempts waste cycles.
If forcing a solo commit: use `git hash-object -w` + `git update-index --cacheinfo`
to bypass WSL stat-cache; avoid `git add` for modified files on WSL/Windows.
