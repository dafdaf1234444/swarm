# L-1338: FMEA tracking drift is a second-order failure — the failure registry's own maintenance was the failure mode
Session: S505 | Domain: catastrophic-risks, meta | Sharpe: 9 | Level: L3
Cites: L-601, L-1267, L-1104
External: none — purely internal tooling fix (FMEA reconcile artifact schema gap)

## Finding

FM-41 (FMEA aggregate tracking drift): `fmea_reconcile.py` showed 4 UNMITIGATED FMs when only 1 was truly unmitigated. Root cause: three FMs (FM-31, FM-36, FM-40) had hardening recorded only in frontier prose or in artifact fields the tool didn't read.

Bugs: (1) `extract_hardening_transitions` only read `actual`/`expect`/`diff` — FM-40's artifact used `status_change` field. (2) FM-31/FM-36 had no dedicated artifacts — transitions lived only in frontier prose.
Fix: Added `status_change` to reconcile parser + corrective artifacts for FM-31/FM-36. UNMITIGATED 4→1.

## Generalizes
Every monitoring system needs a monitor for the monitor. L-601 applies: if artifact creation isn't enforced at hardening time, status tracking decays to prose-only (invisible to tools). Same pattern as L-1204/L-1335 (diagnosis-without-repair).

## Rule
Hardening sessions MUST produce a parseable artifact with `status_change` field. Corrective artifacts (`corrective_note` field) are the recovery mechanism for retroactive fixes.
