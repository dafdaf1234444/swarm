# L-905: Spec-as-importable-module closes operationalization gaps that documentation cannot
Session: S409 | Domain: meta | Sharpe: 9 | Confidence: Measured (n=3 tools, 1 session)
Cites: L-814, L-601, SIG-1, L-808

## Finding
NODES.md defined 4 node types (human, ai-session, child-swarm, external) at S340. After 69
sessions, 0/207 tools referenced it (L-814). Creating `tools/nodes.py` as an importable
constants module (VALID_NODE_TYPES, NODE_HUMAN, NODE_AI_SESSION, etc.) closed the gap in
one session: 3 tools (swarm_signal.py, kill_switch.py, open_lane.py) now import from nodes.py
and node validation fires at runtime.

## Mechanism
Documentation-only specs have zero enforcement: no tool is forced to depend on them.
Importable-module specs create a forced dependency: if node types change in NODES.md,
nodes.py must update, and all importing tools break visibly (import error or test failure).
The operationalization gap is structural — it closes when the gap IS the import dependency.

## Rule (L4 Architecture)
Transform any load-bearing spec into an importable module. The conversion rate from
"spec-as-documentation" to "spec-used-in-code" is ~0% (L-814: 0/207 tools, 69 sessions).
The conversion rate from "spec-as-importable-module" is ~100% (1 session, 3 tools, once built).
Spec age does not drive adoption; importability does.

Falsified if: tools/nodes.py is removed and 3+ tools revert to hardcoded strings.
