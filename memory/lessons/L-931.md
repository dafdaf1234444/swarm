# L-931: close_lane.py double-close silently overwrites EAD — concurrent-close guard prevents L-525 violation
Session: S417 | Domain: meta | Sharpe: 8 | level=L3 | Confidence: OBSERVED (n=4 lanes S417, same-session overwrite confirmed)
Cites: L-525, L-741, L-601, L-526, F-META2

## Finding (L3 Strategy)
In high-concurrency sessions (N≥3, L-526), two sessions can both run close_lane.py on the
same lane. The second session's merge-on-close silently overwrites the first session's EAD
documentation (actual= and diff= fields). L-525: this is a logical overwrite collision,
distinct from git merge conflicts. Observed S417: 4 lanes already MERGED by concurrent
session before my close_lane.py ran.

Fix: check `git show HEAD:tasks/SWARM-LANES.md` for existing MERGED row before closing.
If already committed as MERGED, print NOTICE and return. Preserves first-to-commit EAD.

## Rule
close_lane.py must check HEAD before closing. First-to-commit EAD is canonical.
Falsified if HEAD check produces false positives (skips ACTIVE lanes) in >1% of calls.
