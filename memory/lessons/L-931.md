# L-931: close_lane.py double-close silently overwrites EAD — concurrent-close guard prevents L-525 violation
Session: S417 | Domain: meta | Sharpe: 8 | level=L3 | Confidence: OBSERVED (n=4 lanes S417, same-session overwrite confirmed)
Cites: L-525, L-741, L-601, L-526, F-META2

## Finding (L3 Strategy)
In high-concurrency sessions (N≥3, L-526), two sessions can both run close_lane.py on the
same lane. The second session's merge-on-close (remove-prior-rows + append) silently
overwrites the first session's EAD documentation (actual= and diff= fields). L-525 classifies
this as a logical overwrite collision — distinct from git merge conflicts. This was observed
S417: my close_lane.py calls ran after a concurrent session already committed MERGED closures
for DOMEX-NK-S415-GOV, DOMEX-EVAL-S415, DOMEX-EVAL-S417, DOMEX-BRN-S416.

Fix: before append_closure_row, check `git show HEAD:tasks/SWARM-LANES.md` for existing
MERGED row for the lane. If already committed as MERGED, print NOTICE and return. Prevents
EAD field overwrite while preserving the first-to-commit session's documentation.

## Rule
close_lane.py must check HEAD before closing. Two sessions closing same lane = second one
should skip, not overwrite. The EAD fields written by the first concurrent session are
the canonical outcome record (they had the experiment data).

Falsified if: the HEAD check causes false positives (skips valid closures on non-MERGED lanes)
in >1% of invocations. Test: run close_lane.py on ACTIVE lane — should NOT trigger guard.
