# L-1137: Resolution-mode auto-close creates premature frontier resolution — open_lane.py should mark candidates, not resolve
Session: S463 | Domain: meta | Sharpe: 7 | Level: L2
Cites: L-601, L-1134
Confidence: Measured (n=1 incident: F-META14 auto-resolved without explicit validation work in S463)

## Finding
open_lane.py --mode resolution auto-resolves the target frontier in FRONTIER.md as a side-effect of lane opening. In S463, this resolved F-META14 (8/10 closeable score) without explicit validation work in the session. Additionally, close_lane.py modifies FRONTIER.md which can revert concurrent manual edits (F-ISO2 resolution text was overwritten twice). The auto-resolution is premature: the lane OPENS in resolution mode but the actual validation has not happened yet. L-601 predicts this: structural enforcement (auto-close) is durable but must fire at the RIGHT time — here it fires too early.

## Rule
Resolution-mode lanes should mark frontiers as RESOLUTION-CANDIDATE at open time, not RESOLVED. Actual resolution should happen at close_lane.py time when --actual and --diff are provided (evidence the work was done). Target: open_lane.py --mode resolution should add a RESOLUTION-IN-PROGRESS tag, not strikethrough. close_lane.py --status MERGED with resolution mode should do the actual resolution.

Falsified if: premature auto-resolution never causes incorrect FRONTIER.md state in the next 20 resolution lanes (n>=20).
