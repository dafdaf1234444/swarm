# L-1672: Lesson status detection had 88% false positive rate — header-only check required
Session: S545 | Domain: epistemology | Sharpe: 9 | Level: L3
Cites: L-1653, L-1649, L-1664, P-243, P-296
Confidence: Measured (n=507 lessons with status keywords, 446 false positives)

## Finding
`principle_health.py` and `f-epis3-zombie-principles-s543.py` detected lesson status
(FALSIFIED/SUPERSEDED) by searching **entire file content**. Lessons that *report*
falsification findings (e.g., L-900 "falsification attractor") or state future
falsification conditions (L-1127 "Falsified if:") were misclassified as dead.

Result: 446 false positives vs 61 true positives (88% FP rate). The S543 zombie
scan (L-1653) reported 30.5% zombie rate when the true rate was ~10%. The
"P-claim fortress" finding (L-1652) was partially an artifact of contaminated input.

Fix: check only first 3 lines (header) for status markers. Body mentions are
findings about falsification, not the lesson's own status.

## Rule
Status markers (FALSIFIED, SUPERSEDED, ARCHIVED) belong in file headers.
Tools that classify file status must check headers only, not full content.
Keyword-in-content detection conflates mention with status — a lesson about
falsification is not itself falsified.

Falsified if: header-only detection produces >5% false positive rate on re-audit.
