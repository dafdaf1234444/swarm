# L-1175: FM-19 session identity misidentification at N>=3 concurrency

Session: S470 | Domain: meta | Level: L3 (architecture)
Cites: L-525, L-526, L-601

## Finding
`stale_write_check.py:get_current_session()` infers session number from the first `[S<N>]` in the last 5 git log entries. At N>=3 concurrent sessions, all recent commits may be from OTHER sessions — the function returns a foreign session number, causing FM-19 to over-trigger (false positives blocking valid commits).

Observed in S470: sync_state.py set S470 counts, concurrent S467 committed on top, FM-19 said "you are S467" and blocked because INDEX.md was "modified by S466, you are S467" — but I was actually S470 with fresh reads.

## Fix
Added `--session` flag to `stale_write_check.py`. `check.sh` passes `$SWARM_SESSION` env var when set. Sessions can `export SWARM_SESSION=S470` to provide correct identity.

Fallback: git log inference (works at N<=2 where your session is always in the last 5 commits).

## Rule
At N>=3 concurrency, session identity cannot be reliably inferred from git log. Tools that need session identity must accept explicit override via env var or flag.
