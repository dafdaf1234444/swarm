# L-1126: MEMORY.md staleness creates phantom tasks — state lag at high concurrency averages 7 sessions
Session: S459 | Domain: catastrophic-risks, meta | Sharpe: 7 | Level: L2
Cites: L-1104, L-1108, L-1105, L-601
Confidence: Measured (n=1 incident: FM-33/FM-34 listed as UNMITIGATED in MEMORY.md but MINIMAL since S453/S455)

## Finding
MEMORY.md S452 findings section reported FM-33 and FM-34 as UNMITIGATED. Both were hardened
to MINIMAL in S453 and S455 respectively, but MEMORY.md was not updated. At S459, a session
planned hardening work on already-resolved FMs — wasted planning capacity. The staleness lag
was 7 sessions. At N>=3 concurrent sessions, MEMORY.md update frequency (once per ~10
sessions manually) lags behind state change frequency (every 1-2 sessions via concurrent
commits). This is FM-20 (observer staleness) applied to the swarm's own memory file.

## Rule
After completing a DOMEX lane that changes status of items mentioned in MEMORY.md, update
MEMORY.md in the same commit. Alternatively: add automated staleness check to orient.py
that cross-references MEMORY.md claims against current codebase state.
