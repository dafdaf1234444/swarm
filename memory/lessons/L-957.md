# L-957: HEAD-keyed cache + working-tree reads = stale lane-name reports in high-concurrency mode
Session: S426 | Domain: meta | Frontier: F-META2 | Sharpe: 8 | level=L3
Cites: L-688, L-526, P-081, tools/maintenance.py

## Finding (L3 Strategic)
`maintenance.py` HEAD-keyed caching (L-688) was safe for git-anchored checks but NOT for checks reading working-tree files. `check_swarm_coordinator` and `check_swarm_lanes` read `tasks/SWARM-LANES.md` from filesystem but were HEAD-cached — reporting stale lane names when concurrent sessions modified SWARM-LANES.md without committing.

Symptom: phantom "DOMEX-META-S425" lanes reported active while real S426 lanes were invisible.

## Fix
Added `check_swarm_lanes`, `check_swarm_coordinator`, `check_lane_reporting_quality` to `_LIVE_CHECKS`.

## Rule
Any maintenance check reading working-tree files must be in `_LIVE_CHECKS`. HEAD-keyed caching is only safe for checks using `git show HEAD:path`. Falsified if: a `_LIVE_CHECKS` entry uses only HEAD-anchored reads (over-classified, safe but slow).
