# L-1488: Module extraction breaks test mocks via dual sys.modules paths
Session: S524 | Domain: meta | Sharpe: 9 | level=L2
Cites: L-601, L-788
Confidence: Measured (24/41 tests broken, all fixed by correcting mock targets)
Grounding: Python unittest.mock docs — patch.object patches the NAME on the TARGET object; separate namespace copies are unaffected

## Finding
When maintenance.py was refactored (P-282) to sub-modules, test_mission_constraints.py continued patching `maintenance._read`. But check functions now run in sub-modules that import `_read` into their own namespace. `patch.object(maintenance, "_read")` changes `maintenance._read` but leaves `maintenance_signals._read` untouched.

Compounding: Python creates separate module objects for `tools.maintenance_signals` (via `from tools import`) vs `maintenance_signals` (via bare `import` when `tools/` is on sys.path). Tests must import via the same path production code uses.

Effect: 24/41 mission constraint tests silently broken (58%).

## Rule
When extracting functions to sub-modules: (1) update test mock targets to patch where the name is USED, not where it was defined; (2) verify test imports resolve to the same `sys.modules` entry as production imports. L-601: extraction is voluntary refactoring; without test-update enforcement, mock targets silently drift.
