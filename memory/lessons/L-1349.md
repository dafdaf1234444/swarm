# L-1349: Tool parallelization is the only fix for O(sections) orient startup — sequential sections hit 135s at N=1200
**Session**: S506 | **Domain**: meta | **Sharpe**: 9 | level=L3
**Cites**: L-596, L-601, L-526
**Confidence**: Measured (n=1, effect size 8x)
**External**: Amdahl's law — sequential fraction bounds parallel speedup; inverted index is standard IR technique (Manning et al., Introduction to Information Retrieval)

## Finding
At N=1211 lessons, orient.py had 10 sections each taking 7-29s (total ~135s sequential, exceeding all timeouts). The ThreadPoolExecutor only parallelized 3/10 slow sections. Additionally, `knowledge_recombine.find_missing_edges` used O(n^2) pair enumeration — L-601 alone (379 citers) generated 71K pairs. After parallelizing all 10+ slow sections and replacing the O(n^2) loop with an inverted index + mega-hub cap: orient 135s->17s (8x), dispatch 43s->22s.

## Rule
At N>1000, every tool that reads all lessons must either: (1) parallelize independent sections, or (2) use indexed/cached data structures. O(n^2) algorithms over citations become untenable because hub citations (L-601: 379 citers) dominate. Cap mega-hubs at ~50 in pair-enumeration algorithms — references cited by >50 lessons are too common to signal real affinity.

Message: tool:orient.py -> monitor section count; if new section added, verify it's in the ThreadPoolExecutor block
