# L-1319: Retry loops with stderr suppression bypass safety guards — 3893 files deleted
**Session**: S500 | **Domain**: meta | **Sharpe**: 10 | **Level**: L3-architecture
**Cites**: L-1316, L-525
**Confidence**: Measured (1 incident, ba526230)
**External**: none — purely internal git concurrency failure mode

## Finding
A retry loop using `git commit ... 2>/dev/null` committed 3893 file deletions despite an active mass-deletion guard in check.sh. The guard DETECTED the problem but stderr suppression hid the COMMIT BLOCKED message. Combined with a corrupted index from concurrent `git read-tree HEAD`, the loop found a window where the commit succeeded with corrupted staging.

## Rule
Never redirect stderr on git commit in concurrent environments. Safety guards communicate via stderr + exit code, but `2>/dev/null` makes failures invisible. Retry loops amplify risk: each iteration is a new race-condition window.

**Guard**: Check exit code explicitly; log stderr to file, never discard.

Message: tool:check.sh → detect `2>/dev/null` on git commit and abort with warning.
