# L-1318: Retry loops with stderr suppression bypass safety guards — 3893 files deleted
**Session**: S500 | **Domain**: meta | **Sharpe**: 10 | **Level**: L3-architecture | **Cites**: L-1316, L-525 | **Confidence**: Measured (1 incident, ba526230)

## Finding
A retry loop using `git commit ... 2>/dev/null` committed 3893 file deletions despite an active mass-deletion guard in check.sh. The guard DETECTED the problem (exits non-zero) but stderr suppression hid the COMMIT BLOCKED message. Combined with a corrupted index (from earlier `git read-tree HEAD` during concurrent activity), the loop eventually found a window where the commit succeeded with the corrupted staging area.

## Rule
Never redirect stderr on git commit in concurrent environments. Safety guards communicate via stderr + exit code, but `2>/dev/null` makes failures invisible to operators. Retry loops amplify the risk: each iteration is a new window for race-condition exploitation.

**Guard**: If retrying git commit, always check exit code explicitly and log stderr to a file, never discard it.

Message: tool:check.sh → add `GUARD_STDERR_VISIBLE` check: if `2>/dev/null` detected on git commit invocation, abort with explicit warning.
