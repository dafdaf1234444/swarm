S530 fixed orient.py hangs by wrapping futures in `_safe_result(key, timeout=10)` which catches TimeoutError. But `git_health` and `genesis` futures were left using raw `.result(timeout=10)` — placed before `_safe_result` was defined. On WSL, `git fsck` (30s subprocess timeout) routinely exceeds 10s, causing TimeoutError to crash orient.py.

## Rule
Every future collected from orient.py's ThreadPool MUST use `_safe_result`, never raw `.result()`. When adding timeout wrapping, audit ALL call sites — partial migration creates the illusion of fix while leaving crash paths. `_safe_result` definition should precede all future collection.
