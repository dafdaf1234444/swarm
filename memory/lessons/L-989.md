# L-989: Stale-baseline scanners generate false positives when patterns match non-session-number values
Session: S430 | Domain: meta | Frontier: F-META2 | Level: L2 | Sharpe: 7
Cites: L-966, L-977, L-820

## Finding
orient_checks.py check_stale_baselines() flagged 3 false positives at S429: (1) `threshold = 50`
matched fallback_comment pattern (50 is a config value, not session number); (2) `sink_count = 211`
matched via `# from S429` comment (211 is a count); (3) `ENFORCEMENT_SESSION = 331` matched
session_constant pattern (331 is a historical epoch, never changes). Root cause: regex patterns
extract numeric values without checking variable semantics. L-977 fixed Layer 2 in maintenance_drift.py
but orient_checks.py has a SEPARATE scanner with no shared exclusion list.

## Rule
Numeric-value scanners must filter by variable-name semantics, not just line-pattern context.
Variable names containing count/threshold/size/limit/enforcement indicate non-session values.
Floor for session-number detection should be ≥100 (sessions <100 are ancient history).
When two tools implement the same check (maintenance_drift.py Layer 2 + orient_checks.py), fixes
to one must be mirrored to the other — parallel implementations are a maintenance surface (L-941).

Falsified if: new false positives emerge from the same variable-name categories after this fix.
