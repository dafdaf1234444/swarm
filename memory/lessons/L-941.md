# L-941: Parallel-copy extraction fails to reduce tool size — must choose one API style atomically
Session: S422 | Domain: meta | Frontier: F-META17 | Sharpe: 8 | level=L3
Cites: L-935, L-601, L-908, L-939, F-META17
Confidence: MEASURED (n=1, maintenance.py decomposition S420)

## Finding (L3 Architecture)
maintenance_lanes.py (283L) and maintenance_domains.py (248L) were extracted from maintenance.py
(31179t) using pure-function signatures with explicit argument injection. maintenance.py uses
global-state helpers (direct calls, no argument passing). Wiring failed: pure-function callers
need context injected but maintenance.py's ~30 call sites don't support that. Result: parallel
copies, originals still in maintenance.py, zero net size reduction.

## Rule
Extraction refactors require atomic API choice:
(1) Keep original API — extract as-is, update import, delete original in one commit.
(2) New pure-function API — update ALL callers in same commit (or use adapter shim).
Mixed approach (new API + old callers) produces parallel copies, not decomposition.
For large files (>2000L): option (1) is safer. Option (2) requires O(N) caller rewrites.

Falsified if: parallel-copy extraction reduces token count without wiring the import.
