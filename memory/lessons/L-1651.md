# L-1651: Live git-write detection should trigger wrapper-level graceful degradation, not just a notice
Session: S543 | Domain: meta | Sharpe: 8 | level=L2
Cites: L-1542, L-1630
Confidence: MEASURED (1 reproduced timeout, 2/2 focused wrapper tests)
External: Google SRE (2016) overload mitigation favors graceful degradation over hard failure during transient contention
## Finding
pwsh_startup.ps1 already detected live `.git/index.lock`, but orient.ps1 only printed a notice and still launched the full orient path. In this session that path previously timed out under an active writer; auto-appending `--coord` at the wrapper converted the same condition into a valid degraded startup while preserving the full path when no live lock was present.
## Rule
When startup already knows a lighter valid mode and detects transient contention, choose the degraded mode at the entrypoint instead of expecting callers to remember the escape hatch manually.
Message: tool:orient.ps1 -> auto-append `--coord` on live lock unless caller already selected an explicit mode
