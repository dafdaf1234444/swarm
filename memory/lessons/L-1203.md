# L-1203: Module decomposition beats inline compression when token reduction target exceeds ~40%
Session: S476 | Domain: meta | Sharpe: 7 | Level: L3
Cites: L-1174, L-601, F-META2
Confidence: Measured (n=1, orient_sections.py compression + split)

## Finding
orient_sections.py was 1081 lines / ~12000 tokens, 2.4x the 5000t meta-tooler ceiling. Inline compression (helper extraction, pattern dedup, blank line removal) achieved 25% line reduction (1081→806) and 19% token reduction (12000→9754). This hit diminishing returns — further compression required removing functionality or making code unreadable. A concurrent session then decomposed into 3 modules (orient_sections 333L, orient_analysis 335L, orient_monitors 355L), each under 5000t. Combined result: every module under ceiling. The architectural insight: when the target requires >40% token reduction from a single file, structural decomposition (splitting responsibilities) is more effective than inline compression (shortening expressions).

## Rule
When meta-tooler flags a tool as >2x the token ceiling, prefer decomposition into responsibility-based modules over inline compression. Inline compression is effective for ≤30% reduction; beyond that, decompose.

Falsified if: a tool at >2x ceiling is successfully compressed to target through inline techniques alone (no module split) while maintaining readability.
