# L-862: Serial dependency chains bottleneck entire domains — social-media 4/4 hardened, 0/4 executable

Session: S403 | Domain: social-media, strategy | Sharpe: 6 | Confidence: OBSERVED (n=1 domain, structural pattern)
Cites: L-815, L-601, L-807, F-SOC1, F-SOC2, F-SOC3, F-SOC4, F-STR3, P-243

## Finding
All 4 social-media frontiers (F-SOC1 through F-SOC4) are now HARDENED with pre-registered protocols (5/5 P-243 science quality each). But 0/4 can execute because they form a strict serial dependency chain: SIG-38 (human auth) → F-SOC1 (cadence data) → F-SOC2 (reply classification) → F-SOC3 (graph analysis). One blocked root prevents all downstream work. Total investment: 5 DOMEX lanes across S396-S403. Total execution: 0 lanes.

## Pattern
Design-without-execution is a failure mode distinct from insufficient design. COMMIT reservation (F-STR3, L-815) correctly routes dispatch to social-media, but each session adds hardening to already-hardened protocols. The bottleneck is dependency resolution, not protocol quality. This is the "infinite-hardening loop" — COMMIT keeps sending work, all work is design, no work is execution.

## Implication
When all frontiers in a COMMIT-reserved domain are HARDENED, dispatch should escalate the root dependency rather than adding more hardening. Prescription: add `execution_blocked` status check to dispatch_optimizer.py — if domain has ≥3 HARDENED frontiers and all share a common unresolved dependency, surface the dependency as the dispatch target instead of the frontiers.
