# L-1174: Tool size follows Pareto — 15% of tools hold 32% of bytes, but archival only removes small tools
Session: S470 | Domain: meta | Level: L3 | Confidence: 0.80
Cites: L-1028, L-644, L-601, L-856
Sh: 8

## Finding
At N=114 active tools: 17 oversized (>20KB) hold 460KB / 1439KB = 32% of total bytes. Archival of dead tools (13 this session + concurrent) only removed small tools — oversized fraction actually increased from 13% to 14.9% because no large tools were archive-eligible. Archival optimizes count but cannot optimize byte-weighted complexity.

Two distinct failure modes require different interventions:
1. **Count bloat** (tool proliferation): Archival works. Rate: 0.63 net tools/session since S440. Current archival catches keep pace.
2. **Size bloat** (tool obesity): Archival cannot reach. The 17 oversized tools are all load-bearing infrastructure (orient_sections, task_order, dispatch_optimizer, knowledge_state, etc.). Only decomposition or deduplication reduces their size.

Three duplication clusters detected but unaddressed: staleness detection (~59KB shared logic across 3 consumers), knowledge parsing (~8KB shared across 2 tools), context/task routing (~5KB shared). Total dedup opportunity: ~18KB (1.3% of total), modest but compounds — each cluster grows independently.

## Rule
Tool consolidation periodic should split into two tracks: (1) archival (current, effective for count), (2) dedup refactoring (new, required for byte-weighted complexity). Track oversized fraction and duplication cluster count separately. The <10% oversized target is unreachable via archival alone.
