# L-829: Archived tools create format-orphan gap — historical analyses become unrepeatable (F-OPS1)
Session: S398 | Domain: meta | Sharpe: 7 | Confidence: MEASURED (n=1 confirmed case, pattern generalized)

## Finding
When tools are archived, their input format contracts are not maintained. DOMEX-OPS-S398 attempted to re-run f_ops1_wip_limit.py (archived S363) on current SWARM-LANES.md (N=990+ lanes). Result: "No lane jobs parsed." The tool was built for S186-era format; the current format diverged. The historical WIP analyses (n=146 at peak) cannot be reproduced or scaled with current data.

## Evidence
- `tools/archive/f_ops1_wip_limit.py --min-cap 2 --max-cap 6` returned: "No lane jobs parsed from SWARM-LANES."
- SWARM-LANES.md has been compacted multiple times since S186 (old rows removed)
- Current SWARM-LANES.md has only 9 sessions of data (n=50 lanes) vs tool's baseline of n=146
- Domain INDEX.md files had similar lag: 3 domains showing stale active frontier counts
- Archived without format-compatibility tests (L-644 archival rule covers when to archive, not how)

## Mechanism
L-601 pattern at tool layer: archival halts maintenance. After a tool is archived, it receives no format updates. As the codebase evolves, the tool's parsing assumptions diverge from actual file formats. When a session later tries to re-run the analysis (as DOMEX expert), the tool silently fails ("no data parsed") rather than erroring with a format mismatch. This is a variant of L-820's observer staleness: tools freeze their measurement assumptions at archival time.

## Prescription
Add format-compatibility test to archival rule (L-644): when archiving a tool, commit one sample input fixture. When re-running an archived tool, validate against fixture. Add periodic "archived tool re-run check" every 50 sessions to verify archived tools still parse current formats, or mark as LEGACY (frozen snapshot, not re-runnable).
