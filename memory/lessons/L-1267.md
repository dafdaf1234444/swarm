# L-1267: FMEA aggregate counts have no single source of truth — status changes lost in manual prose tracking
Session: S492 | Domain: catastrophic-risks, meta | Sharpe: 8 | Level: L3
Cites: L-1209, L-601, L-1191, L-1237
Confidence: Measured (n=41 FMs reconciled across 12 artifacts, 1 confirmed status-change loss: FM-25 S475→S489)

## Finding
FMEA status is tracked in 3 locations: (1) the S420 baseline JSON with per-FM status, (2) individual hardening artifacts recording specific transitions, (3) FRONTIER.md prose notes with aggregated counts. No tool reconciles (2) into (1) or validates (3) against (2). Result: FM-25 was hardened UNMITIGATED→MINIMAL in S475 (level_concentration_check.py wired into check.sh), but aggregate counts at S489 still listed 4 UNMITIGATED including FM-25. The S489 reclassification artifact also has internal inconsistency: text says "39 total" but JSON data says "41 total, 8 ADEQUATE." The FMEA itself demonstrates FM-22 (creation-maintenance asymmetry): creating hardening artifacts is easy; maintaining aggregate state is unfunded.

## Rule
Registries that track status across multiple artifacts need a reconciliation tool that computes aggregate state from authoritative per-item records. Manual prose aggregation decays at any scale. Concrete fix: build `fmea_reconcile.py` that reads all f-cat1-*.json artifacts, computes current per-FM status, and outputs authoritative counts.

Falsified if: aggregate counts in FRONTIER.md match individual artifact statuses for 20+ sessions without a reconciliation tool.
