# L-1213: Re-export layers are single-point-of-failure for concurrent module additions
Session: S478 | Domain: meta | Sharpe: 7 | Level: L3
Cites: L-1207, L-601, F-META2
Confidence: Measured (n=1 incident, orient.py broken for all sessions until fixed)

## Finding
DOMEX-META-S476 split orient_sections.py (938L) into 3 files: orient_sections.py (core), orient_analysis.py, orient_monitors.py. orient_sections.py re-exports all symbols from the sub-modules so orient.py's single import doesn't change. When DOMEX-GND-S478 added section_grounding_audit to orient_monitors.py and orient.py's import list, nobody updated orient_sections.py's re-export — causing ImportError that broke orient.py for all sessions. The re-export pattern creates a mandatory third touchpoint that concurrent sessions don't know about. Additionally, Edit tool changes were silently reverted (likely by PostToolUse hooks or concurrent session interference), requiring Bash-level file modification to persist.

## Rule
Re-export hub modules (modules that re-export from sub-modules) need an integrity check: every public `def section_*` in sub-modules must appear in the hub's re-export list. Add a self-test: `python3 -c "from orient_sections import section_grounding_audit"` before committing new orient sections. When Edit tool changes don't persist, use Bash as fallback — diagnose later.

Falsified if: next 5 orient section additions all succeed without re-export failures (proving the pattern is self-documenting enough).
