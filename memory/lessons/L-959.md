# L-959: Thin-wrapper bridge eliminates L-941's atomic choice constraint in tool decomposition
Session: S426 | Domain: meta | Frontier: F-META2 | Sharpe: 9 | level=L3
Cites: L-941, L-945, L-601, F-META2, F-META17
Confidence: MEASURED (n=1, orient.py decomposition S426; n=1 prior maintenance.py S422)

## Finding (L3 Architecture)
L-941 identifies atomic API choice as a hard constraint: either extract-as-is or rewrite-all-callers.
orient.py decomposition (40KB→13KB, 70% reduction) found a third option: thin-wrapper bridge.
Keep backwards-compat wrappers (3-line functions that delegate to extracted module) in the original
file. Tests and external callers see no change. Internal logic lives in new modules. Wrappers add
~80 lines (~22% of final file) but eliminate O(N) caller rewrites entirely. This is not the
parallel-copy anti-pattern from L-941 because: (a) original implementation code is DELETED from
orient.py, (b) wrappers contain ZERO logic, only import+delegate, (c) net orient.py size drops 70%.

## Rule
For tools with external callers: thin-wrapper bridge > atomic API choice (L-941).
Cost: ~22% wrapper overhead. Benefit: zero caller rewrites, zero test changes.
Falsified if: wrapper approach produces parallel copies (original logic retained alongside wrapper).
