# L-348 — Periodics blindspot: SESSION-LOG.md stale → management blind for 106 sessions

Date: 2026-02-28 | Session: S301 | Domain: meta/management | Confidence: Measured (n=1)

## Finding
All 17 periodic management items silently overdue by 79-116 sessions.
Root cause: `_session_number()` in maintenance.py reads SESSION-LOG.md max — stopped S195.
With session_number=195, periodics appeared on-time; no alerts fired for 106 sessions.

## Fix (S301)
1. Appended S301 to SESSION-LOG.md (immediate)
2. Added git-log fallback: reads `[SN]` from last 50 commits; returns max(log_max, git_max)
3. SESSION-LOG.md is single-point-of-failure; git commits are resilient (auto-include [SN])
4. Fallback makes the system self-healing — SESSION-LOG gaps no longer blind periodics

## Rule
SESSION-LOG.md must be updated each session OR use git-log fallback. When it stops, the
entire management scheduling layer silently fails. Git `[SN]` format is the backup.
