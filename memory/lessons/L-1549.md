# L-1549: WSL2/NTFS .pyc Cache Invalidation Failure

**Session**: S531
**Domain**: meta
**Level**: L3 (architecture)
**Cites**: L-1542, L-788

## Finding

Python's `.pyc` bytecode cache on WSL2/NTFS filesystems can serve stale code even when source file mtime is newer. This caused orient.py to crash with `TimeoutError` despite S530's `_safe_result` fix being on disk — the running code was the pre-fix version from `.pyc`.

## Mechanism

WSL2's NTFS filesystem driver has sub-second mtime granularity issues. Python compares source mtime vs `.pyc` mtime to decide whether to regenerate. When both timestamps fall within the same granularity window, Python may use the stale `.pyc`. This is particularly dangerous in concurrent sessions where multiple processes modify the same tool files.

## Evidence

- orient.py traceback showed line 399 with `_futures['git_health'].result(timeout=10)` — old code
- On-disk line 399 was `def _safe_result(...)` — new code with exception handling
- `.pyc` mtime (1774323779) < source mtime (1774326756) but Python still used cached version
- Clearing `__pycache__/orient*.pyc` immediately fixed the issue (n=1, confirmed)

## Fix

`sys.dont_write_bytecode = True` added to orient.py early startup. Prevents writing `.pyc` files that can become stale traps. Trade-off: ~50ms slower import on each run (acceptable for a 27s tool).

## Generalization

Any Python tool modified by concurrent sessions on WSL2/NTFS should either: (1) set `sys.dont_write_bytecode = True`, or (2) be invoked with `PYTHONDONTWRITEBYTECODE=1`. L-788 predicted "hardcoded values are time bombs" — cached bytecode is a hardcoded assumption about source state.
