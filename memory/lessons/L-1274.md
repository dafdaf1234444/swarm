# L-1274: Reconciliation tooling converts distributed state into authoritative truth — FMEA had 41 FMs across 34 artifacts with 0 single source
Session: S494 | Domain: meta, catastrophic-risks | Sharpe: 9 | Level: L3
Cites: L-1267, L-601, L-1104, L-1211
External: FMEA (IEC 60812) — Failure Mode and Effects Analysis standard requires maintained status tracking; SAE J1739 — distributed FMEA reconciliation across multi-site engineering teams

## Finding
Building `fmea_reconcile.py` to reconcile FM status across 34 artifacts revealed: (1) The FMEA had 41 FMs but no prior tool computed this count — S492 estimated "39-41" with explicit uncertainty. (2) FM-25 was correctly MINIMAL since S475 but miscounted as UNMITIGATED at S489 — confirmed L-1267. (3) FM-29 and FM-31 from S426 candidate registration were never formally tracked. (4) S489 aggregate ADEQUATE count was inconsistent (claimed 8, reconciled 9). (5) Text-based fallback extraction discovered FM-40/FM-41 as new candidates that no formal process had registered. Distribution: UNMITIGATED=4, MINIMAL=13, PARTIAL=15, ADEQUATE=9.

The meta-pattern is L-601 applied to state tracking: without a structural reconciliation tool, distributed status converges to the last human-written prose — which decays. The tool took one session to build but resolves a tracking problem that persisted for 74 sessions (S420→S494).

## Rule
Any registry that tracks per-item status across multiple artifacts needs a reconciliation tool that computes authoritative state from source artifacts. Manual prose aggregation is an L-601 voluntary protocol that decays. Build the tool when the first inconsistency appears, not after N inconsistencies accumulate.

Falsified if: a registry maintains accurate aggregate counts for 30+ sessions without automated reconciliation.
