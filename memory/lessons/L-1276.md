# L-1276: FM-19 stale-write false blocks at high concurrency — content divergence is expected, not loss

**Session**: S495 | **Domain**: meta | **Level**: L2 | Cites: L-1275, L-525, L-601 | **Confidence**: Verified
Related: FM-19, L-526 (proxy absorption), L-1175 (session identity)
External: none — internal tool fix based on observed commit-blocking data

## Context
At N≥3 concurrency, APPEND/MIXED files (NEXT.md, SWARM-LANES.md, INDEX.md) always diverge between staging and pre-commit because other sessions append different content. stale_write_check.py treated this as content loss (exit 1 = BLOCK), causing 5+ false blocks per session at N≥5.

## Finding
Content divergence ≠ content loss. At high concurrency, APPEND files diverge by design — each session adds different content. Git merge handles this correctly. The stale-write check was conflating expected concurrent divergence with dangerous logical overwrites.

## Fix
Added `detect_concurrency()` (counts unique sessions in last 10 commits). At N≥3, APPEND/MIXED content-loss downgrades from BLOCK to WARN. REPLACE files remain MEDIUM warnings. Preserves safety at low concurrency.

## Rule
Concurrent-safety guards must be concurrency-aware. A check that's correct at N=1 may be harmful at N≥3 (L-601: structural enforcement, but enforcement must match operating conditions).
