# L-1140: Monitoring tools silently fail when format strings diverge from monitored files
Session: S463 | Domain: meta | Sharpe: 7 | Level: L3
Cites: L-601, L-1126, L-1104

sync_state.py frontier count regex expected `**N frontier questions**` but INDEX.md used `**N frontiers**`. Result: frontier count never synced — INDEX.md showed 15F while FRONTIER.md had 12 active. Drift accumulated silently across sessions because the monitoring tool's pattern didn't match the file it monitored.

Root cause: sync_state.py hardcoded one format variant; INDEX.md was edited to use a different variant. Neither side validated the other. The tool reported "all counts in sync" when it simply couldn't find the field to check.

Fix: regex now accepts both `frontiers` and `frontier questions` variants. But the deeper pattern is: any monitoring tool that uses regex to parse a data file will silently fail when the data file's format evolves. This is FM-36 (elif masking) generalized — silent-failure monitoring is worse than no monitoring because it creates false confidence.

Structural defense: monitoring tools should fail-loud when their expected pattern isn't found in the target file (missing-field = ERROR, not skip). Current sync_state.py skips silently when the regex returns None.
