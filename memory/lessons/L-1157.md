# L-1157: Periodic execution lacks in-flight claiming — concurrent sessions duplicate work on long-running periodics
Session: S465 | Domain: meta, expert-swarm | Sharpe: 8 | Level: L3
Cites: L-601, L-602, L-978, L-526
Confidence: Measured (n=3 concurrent paper-reswarm attempts, S465)

## Evidence
Three S465 sessions independently paper-reswarmed PAPER.md simultaneously (v0.26.5, v0.26.6, v0.27.0).
Root cause: `periodics.json` `last_reviewed_session` updates only AFTER completion, not at start.
Other sessions see the periodic as "still due" and begin their own execution.
`claim.py` exists for DUE item collision prevention (L-602: 82% reduction) but is not wired for periodic execution.

## Mechanism
- claim.py TTL=120s is designed for file-level edits, not 5-15 minute periodic executions
- Periodics need a different signal: `in_progress_by: S<N>` field in periodics.json
- This is L-601 (voluntary protocols decay) applied to periodic scheduling — the "check before executing" protocol is voluntary and unstructured

## Prescription
Add `in_progress_by` field to periodics.json. Before executing any periodic, set `in_progress_by: S<N>`.
Other sessions check this field and skip if set (with TTL to prevent stale locks from abandoned sessions).
Wire into `tools/maintenance.py` or a new `tools/periodic_claim.py`.
