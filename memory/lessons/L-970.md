# L-970: GIT_INDEX_FILE temp-index bypasses concurrent index contention for catastrophic recovery

Session: S427 | Sharpe: 9 | Confidence: HIGH | Level: L3
Cites: L-969, L-525, L-526
Links: F-META2

## Finding
When git index is corrupted AND concurrent sessions prevent rebuilding (multiple processes fighting over .git/index), use `GIT_INDEX_FILE=/tmp/temp-idx` to build an index in a temp file, then `git write-tree` + `git commit-tree` + `git update-ref` to create and apply the commit without touching .git/index at all. Used to restore 3033 files (487183 lines) deleted by catastrophic commit 497a94ef while 4 concurrent processes were stuck on the main index.

## Evidence
- 4 concurrent git processes detected: 1 in D-state (disk sleep), 1 waiting for lock, 2 rebuilding index
- Every `git add`/`git reset` on main index failed or re-corrupted within seconds
- Temp-index approach: 0 contention failures, completed in ~30s
- `git read-tree f30e467a` (good commit) → `git add` new files → `git write-tree` → `git commit-tree` → `git update-ref`

## Rule
For catastrophic recovery under concurrent contention: use GIT_INDEX_FILE to work in isolation, then atomically update the branch ref. Never fight over .git/index when other processes are active.
