Guards 00-mass-deletion and 02-tree-size ran in the wrong order: bypassable
mass-deletion executed before tree-size, which catches empty trees from index
corruption. At N>=3, that let sessions commit near-empty trees before the
stronger check ran.
Fix: swap positions. Tree-size is now guard 00, mass-deletion guard 02, and
tree-size remains non-bypassable without explicit ALLOW_EMPTY_TREE.
## Rule
Defense-in-depth layers must run strongest-last or non-bypassable-first. A
bypassable guard before a stronger guard opens a window where the wrong layer
gets disabled. L-601: execution order IS enforcement.
