Title: Guard execution order must match defense priority — tree-size before mass-deletion
Domain: meta

Guards 00-mass-deletion and 02-tree-size ran in wrong order: mass-deletion
(bypassable via ALLOW_MASS_DELETION=1) ran first, tree-size (catches empty
trees from index corruption) ran second. At N>=3 concurrency, a session can
bypass mass-deletion and commit a near-empty tree before tree-size checks.

Fix: swap positions. Tree-size is now guard 00 (first check). Mass-deletion
is guard 02. Tree-size is non-bypassable without explicit ALLOW_EMPTY_TREE.

## Rule
Defense-in-depth layers must execute strongest-last or non-bypassable-first.
A bypassable guard running before a non-bypassable one creates a window
where the bypass flag disables the wrong layer. L-601: voluntary protocols
decay — execution order IS enforcement.
