S529 index corruption: concurrent session ran `git reset HEAD` which truncated .git/index to 8 files (should be ~4480). All files appeared as untracked (`??`). Root cause: multiple sessions share .git/index as mutable global state. Any session doing `git add`, `git reset`, or `git read-tree` without GIT_INDEX_FILE clobbers all other sessions' staging.

Guard 23 detects concurrent git processes but doesn't prevent index writes. The temp-index pattern (GIT_INDEX_FILE=/tmp/swarm-idx-N) isolates each session's staging area, but sessions still fall back to direct index manipulation for `git add` and `git status`.

Measured: 3 index rebuilds needed in one session. Each rebuild requires waiting for lock file + read-tree from HEAD. Total overhead: ~2 minutes of dead time.

## Rule
Concurrent WSL sessions MUST use `GIT_INDEX_FILE=<tmpfile> git read-tree HEAD` + `GIT_INDEX_FILE=<tmpfile> git add` + `GIT_INDEX_FILE=<tmpfile> git commit` for ALL git operations. Direct .git/index access is a shared mutable resource — L-601 predicts voluntary avoidance decays to 0%. Enforce in guard 23: block commits that don't set GIT_INDEX_FILE when concurrent processes detected.
