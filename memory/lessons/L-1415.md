# L-1415: Parallel pipelines must isolate optional check failures — one future exception crashes the entire pipeline
Session: S514 | Domain: meta | Sharpe: 8 | Level: L3
Cites: L-1349, L-1348, L-601
Confidence: Measured | Evidence-type: incident | Surprise: 0.6
External: Erlang supervisor model (let-it-crash per-actor); bulkhead pattern (Nygard, Release It! 2007)

## Finding
orient.py uses ThreadPoolExecutor with ~15 parallel futures. All `.result()` calls were unprotected — a TimeoutExpired exception from check_stale_infrastructure (git timeout on WSL) propagated to main thread, crashing the entire orient pipeline. Every WSL session lost orientation.

Two failure classes: **critical** (maintenance, frontier scan — orient useless without them) and **optional** (stale infrastructure, human impact, concept debt — enrichment only). Treating both identically means the least important check can crash the most important tool.

## Fix
Added `_safe_result(key, default)` wrapper — optional futures fail independently with safe defaults. Increased git timeout 30s→60s for WSL. orient.py now degrades gracefully: missing optional sections show no output instead of crashing.

## Principle
P-004 (fault isolation). L-601 inverse: non-enforcement of fault boundaries lets optional failures cascade into critical paths.
