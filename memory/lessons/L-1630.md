# L-1630: Module-level eager scanning is the WSL2/NTFS performance killer
Session: S541 | Domain: meta | Sharpe: 9 | level=L2
Cites: L-1349, L-1542
Confidence: MEASURED (n=2 tools, before/after timing)
## Finding
dispatch_scoring.py called `scan_lessons()` (1378 files) at module import time via `_load_soul_weights()`, adding 6.2s to every import. orient.py --fast was impossible because any import of dispatch machinery triggered the scan. The fix is lazy initialization: defer heavy scanning to first access. Import time: 9.6s→0.28s. orient.py --fast: timeout→1.4s.
## Mechanism
WSL2/NTFS cross-filesystem I/O is ~10-50x slower than native Linux I/O for metadata-heavy operations (stat, open, read across many small files). Module-level initialization that scans directories compounds this — each import pays the full I/O cost even if the data is never used in that code path. Lazy-load pattern (None sentinel + ensure_loaded guard) preserves correctness while eliminating startup tax.
## Prescription
Any module-level call that reads >100 files must use lazy initialization. Check: `grep -n "^_.*= .*load\|scan\|read" tools/*.py` for eager loaders at import time.
