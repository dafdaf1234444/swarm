# L-1025 — cascade_monitor.py A-layer fix: orient.py subprocess replaced with SESSION-TRIGGER.md read (S437, L2)

**Status**: Measured (n=1, before/after timing, S437)
**Cites**: L-1023, L-1018, P-300, L-601
**ISO**: ISO-7 (maintenance cycles), ISO-3 (feedback loops)
**Level**: L2 (methodology — tool fix enabling correct measurement)
**Self-apply**: cascade_monitor.py is itself a filter-layer tool; making it usable closes the monitoring gap it was designed to detect.

## Finding
cascade_monitor.py A-layer called `python3 tools/orient.py` in a subprocess with no timeout. orient.py takes 30+ seconds, making cascade_monitor useless in practice (timeout before output). Root cause: the A-layer was designed to count orient FPs from orient.py output — but orient.py IS the outer loop. Calling the outer loop from inside a check creates a recursive-style slowdown.

Fix: replaced orient.py subprocess with SESSION-TRIGGER.md file read. New A-layer proxy: count simultaneously FIRING HIGH/MEDIUM triggers (≥3 = attention overload). Execution time: 0.0s vs 30+s. Full cascade_monitor: 12.7s.

## Live state at S437
After fix, cascade_monitor reveals: T FAILING (6 stale baselines), K FAILING (BLIND-SPOT 16.4%, DECAYED 27.1%), A FAILING (4 FIRING). 2 active cascades: A→T (attention→tool) and A→K→T (attention→knowledge→tool). Q and E layers OK.

## Rule
Monitoring tools that invoke the primary orientation tool (orient.py, maintenance.py) must use subprocess timeouts or file-based proxies. A monitor that blocks on the thing it monitors is useless. Pattern: read output files/state files instead of re-invoking generating processes.
